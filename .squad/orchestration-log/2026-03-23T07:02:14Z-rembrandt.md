### Work Summary

Designed the Scout beacon redesign:
- Beacon as passive role trait (always-on for Scout when roles enabled)
- Share Intel as explicit Scout ability with 60s cooldown
- Auto-activation trigger: `GameplayService.UpdatePlayerLocation` when Scout has valid GPS
- Guards on `ActivateBeacon`/`DeactivateBeacon` for Scouts (prevent manual toggle)
- Cone computation remains server-side (via `VisibilityService.ComputeBeaconSectorKeys`)
- Non-role-mode games unaffected (backward compatible)

### Architecture Decisions

1. **Auto-activation hook:** `UpdatePlayerLocation` is the right place because:
   - Beacon already requires valid location — auto-activation precondition is implicit
   - Already maintains `BeaconLat/Lng/Heading` when `player.IsBeacon`
   - Reconnecting Scouts recover naturally via persisted `PlayerDto.IsBeacon`

2. **Hub method consolidation:** No new methods needed
   - `ShareBeaconIntel` remains hub-exposed
   - Backend ignores client-supplied hex list, computes cone server-side
   - Cooldown (`PlayerDto.ShareIntelCooldownUntil`) enforced at service level

3. **Role awareness:** All backend/frontend changes gated on `PlayerRolesEnabled`
   - Non-role mode preserves existing toggle behavior unchanged

### Frontend Implications

- Replace beacon toggle with "Share Intel" pill for Scouts
- New `ShareIntelCard` component (cooldown timer + CTA)
- `AbilityOverlayLayer` cone rendering unchanged (relies on `player.isBeacon` which is now auto-set)
- i18n keys: `abilities.beacon.alwaysOn`, `abilities.beacon.waitingForGps`, `abilities.shareIntel.*`
