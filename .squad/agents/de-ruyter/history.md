# De Ruyter — History

## Core Context
Backend Dev on Landgrab. ASP.NET Core 8, SignalR, EF Core, SQL Server/PostgreSQL. All game state is in-memory (singletons). GameService is the main façade. Hub split into 4 partial classes.

Key patterns:
- Endpoint extension methods on WebApplication
- JWT: ClaimTypes.NameIdentifier + JwtRegisteredClaimNames.Sub both present
- BCrypt factor 12, password min 8 chars
- Rate limiting "auth" policy on /api/auth/*
- AllianceDto (transient) ≠ Alliance EF entity (persistent)

## Learnings
- Team hired 2026-03-22 by Léon van de Broek
- 2026-03-22: Setup wizard location gating can get stuck due to frontend-side race (SetWizardStep(1) sent optimistically before SetMapLocation state update lands). Backend accepted manual coordinates already (lat/lng only), but wizard progression depended on subsequent state timing. Added backend guard in MapAreaService.SetMapLocation: when map location is set while CurrentWizardStep == 0, auto-advance to step 1 in the same authoritative state snapshot. This keeps manual coordinate flow and GPS flow both deterministic without changing SignalR message shape.
- **2026-03-22 (steen-continued-ux cross-reference):** Wizard fix was validated in 6-player playtest, but downstream gameplay reveals 4 critical/major blockers that require follow-up: null currentHex on game start, no debug movement fallback, false-success action feedback, no in-game location recovery. See .squad/decisions.md items 4–6.
- 2026-03-22: Fixed null `currentHex` at game start by initializing each player's position during `LobbyService.StartGame`. Spawn selection is deterministic and uses priority order: player-owned tile → alliance-owned tile → master tile → nearest grid fallback; this sets `CurrentLat/CurrentLng` and `CurrentHexQ/CurrentHexR` before entering Playing so location-gated actions no longer fail on first interaction. Kept SignalR payload shape unchanged and preserved existing mobile GPS + desktop keyboard update paths.
- 2026-03-22: Fixed manual HQ assignment consistency and validation in `AllianceConfigService.SetAllianceHQ`. The service now enforces alliance-ownership for the selected HQ tile, then appends an `AllianceHQAssigned` event **after** state assignment using the same `(q,r)` values written to `alliance.HQHexQ/HQHexR`, removing event/state mismatch risk for manual HQ placement. Added tests for success event payload and non-owned tile rejection.
- 2026-03-22: Investigated Q/E heading behavior from keyboard playtest. Backend hub methods that consume heading (`ActivateBeacon`, `ResolveRaidTarget`, `ResolveTacticalStrikeTarget`, `AttemptIntercept`) exist and validate heading; issue is frontend interaction gating: Q/E rotation handler exists in `GameMap.tsx` but only runs when compass rotation mode is enabled via the compass button.
- 2026-03-22: Investigated Dutch locale event log issue. Backend emits mostly hardcoded English `GameEventLogEntry.Message` strings across services. Frontend localizes only known `event.type` values and falls back to raw server `message` for unknown types, so many entries remain English in Dutch UI. Recommended direction: adopt typed event-key contract and localize on frontend; keep server `Message` as optional diagnostic fallback only. See .squad/decisions.md item 16–19.
- 2026-03-22: Beacon intel sharing now reuses a single backend sector computation path. Extracted `VisibilityService.ComputeBeaconSectorKeys(GameState state, PlayerDto player)` and consumed it from both fog-of-war visibility and new `AbilityService.ShareBeaconIntel(roomCode, userId)`, preventing drift between what a scout sees and what alliance intel-sharing snapshots persist.
- 2026-03-22: Reworked `VisibilityService` intel model to make beacon sector visibility personal-only and preserve Scout-gated sharing through explicit `ShareBeaconIntel`. `ComputeVisibleHexKeys` now adds beacon sector keys only for the viewing player when `IsBeacon` is true, and adds always-fresh alliance-border enemy tiles by scanning neighbors of alliance-owned hexes. `UpdateMemory` now writes all visible hostile intel to viewer memory but auto-shares only `visibleHostileKeys - beaconSectorKeys` to allied memories, so beacon-derived intel is never passively broadcast while border/proximity intel still is. (Cross-ref: vondel-intel-design produced 5 design options, Eyes of the Scout selected; de-ruyter-visibility implemented with tests: 289 total, 288 passed, 1 skipped.)
- 2026-03-22: Fixed beacon reveal pipeline so active beacons re-scan on every `UpdatePlayerLocation` (movement and heading changes) and publish scan geometry for UI overlays. Root cause: beacon sector intel only lived in computed visibility during full `StateUpdated`, but movement updates often sent `PlayersMoved` only, so no fresh projection/memory update happened while beacon moved/rotated. Added `PlayerDto.BeaconScanHexes` (grid keys) and now refresh `BeaconLat/BeaconLng/BeaconHeading/BeaconScanHexes` in `GameplayService.UpdatePlayerLocation` when `IsBeacon` is active, forcing `StateUpdated` broadcast via `gridChanged`. Also ensured snapshot cloning carries `HexCell.LastSeenAt` so remembered enemy intel staleness survives projection for Amber Archive.
- 2026-03-22: Adopted full-tile broadcast model for fog-of-war. `VisibilityService.BuildStateForViewer` no longer masks hidden cells; it only sets `VisibilityTier` while preserving tile gameplay fields. Removed backend-only beacon scan geometry (`PlayerDto.BeaconScanHexes`) and movement-triggered beacon re-scan/grid-change forcing. Added explicit share flow `ShareBeaconIntel(roomCode, hexKeys)` so scout clients send cone hex keys and backend refreshes `PlayerVisibilityMemory.RememberedHexes[*].SeenAt` for alliance members. Validation: `dotnet build` + `dotnet test` passing (293 total, 292 passed, 1 skipped).
- **2026-03-22 (beacon-architecture-simplification):** Completed architectural refactor with vermeer. Removed server-side masking (`VisibilityService.BuildStateForViewer` only sets `VisibilityTier`), removed implicit beacon re-scan on movement (`GameplayService.UpdatePlayerLocation` no longer forces `gridChanged` from beacon heading/position), deleted `PlayerDto.BeaconScanHexes` field. Backend now exposes hub method `ShareBeaconIntel(roomCode, hexKeys[])` that receives cone hex list from client and updates alliance `PlayerVisibilityMemory.RememberedHexes` with fresh `SeenAt` timestamps. Frontend computes cone geometry client-side via new `beaconCone.ts` utility and invokes ShareBeaconIntel explicitly. Rationale: aligns backend role (authoritative state) with display policy (client rendering). Eliminates redundant server cone computation. Clarifies scout teamwork as explicit action. Validation: `dotnet build` ✅, `dotnet test` ✅ (292/293, 1 skipped). See orchestration log 2026-03-22T17:39:41Z-de-ruyter.md.
- 2026-03-23: Redesigned Scout beacon backend flow so beacon is role-passive (always on for Scout when roles are enabled) and Share Intel is the active Scout ability with a 60s cooldown. Added `PlayerDto.ShareIntelCooldownUntil`, centralized scout beacon state sync via `GameStateCommon.SyncBeaconStateForRole(state, player)`, and updated role lifecycle paths (`LobbyService`, `RoomService`, `GameplayService` movement sync) so scout `IsBeacon/BeaconLat/BeaconLng/BeaconHeading` stay authoritative without requiring deactivate/activate toggles. `AbilityService.ShareBeaconIntel` now ignores client-supplied hex lists and computes the cone server-side through `VisibilityService.ComputeBeaconSectorKeys`, then writes remembered intel for alliance members with existing staleness semantics. Validation: `dotnet build --configuration Debug` (Landgrab.Api) ✅ and `dotnet test` (Landgrab.Tests) ✅ (295 total, 294 passed, 1 skipped).
- 2026-03-23: Fixed delayed proximity reveal after movement by updating `GameHub.UpdatePlayerLocation` to broadcast full `StateUpdated` whenever the mover changes hex (`PreviousHexKey` vs current key), not only when `gridChanged` is true. This forces immediate per-viewer visibility recomputation through `VisibilityBroadcastHelper.CreateStateForViewer` + `VisibilityService.ComputeVisibleHexKeys` so non-Scout 1-hex adjacency reveals appear instantly while Scout beacon behavior remains unchanged. See `.squad/orchestration-log/2026-03-23T13:14:04Z-de-ruyter.md` and `.squad/decisions.md`.
- 2026-03-23: Optimized movement broadcasts to eliminate unnecessary full-state recomputation. With frontend now deriving visibility locally, backend no longer broadcasts full `StateUpdated` on every hex boundary crossing. Changes: (1) `GameplayService.UpdatePlayerLocation` no longer sets `gridChanged = true` when updating Shepherd `LastVisitedAt` metadata — this value is server-side only for `TroopRegenerationService` decay logic and doesn't need immediate client broadcast. (2) `GameHub.UpdatePlayerLocation` now only calls `BroadcastState` (full per-viewer projection) when `gridChanged` is true (territory claims, forts, combat, phase changes). Simple position updates use lightweight `BroadcastPlayersPerViewer` instead. Removed unused `HasPlayerChangedHex` helper. Validation: `dotnet build` ✅, `dotnet test` ✅ (295 total, 294 passed, 1 skipped). Rationale: Reduces O(N×M) visibility recomputation on every movement; frontend computes visibility from player positions client-side.
