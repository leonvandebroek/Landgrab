2. CORE ARCHITECTURE
═══════════════════════════════════════════════════════════════════════════════

DATA FLOW:
  1. SignalR receives grid update → onStateUpdated handler
  2. normalizeGameState() called → useGameStore.setGameState()
  3. useMapOrchestrator detects changes (via useGridDiff)
  4. Movements/events synced to effectsStore (visual effects)
  5. Tile state synced to tileOverlayStore (display state)
  6. GameMap renders tileOverlayStore + effectsStore visuals

KEY HOOKS (in order of execution):
  1. useSignalR — WebSocket connection (maintains hub connection)
  2. useSignalRHandlers — Event handlers (routes server messages)
  3. useMapOrchestrator [NEW] — Phase 3 coordination (orchestrates sub-hooks)
     ├─ useGridDiff — Movement detection (returns TroopMovement[])
     └─ useEffect chains — Store syncing + event tracking
  4. GameView — Presenter (reads stores, renders components)

HEX COORDINATE KEY FORMAT:
  ✓ Correct: "${q},${r}" (e.g., "5,-3")
  ✗ Wrong: "q:r", "q_r", "q.r", etc.
  Functions: parseKey(key) and toKey(q, r) exist in useGridDiff.ts

4. CRITICAL GOTCHAS & WARNINGS
═══════════════════════════════════════════════════════════════════════════════

1. HEX KEY FORMAT IS "${q},${r}"
   ✗ Wrong: "5:3", "5_3", "5|3"
   ✓ Right: "5,-3"
   Impact: All lookups fail if format wrong

2. useGridDiff TIMING
   ✗ First render has NO previous grid — returns empty movements
   ✓ Only second+ update triggers detection
   Impact: Don't expect movements on initial load

3. useGridDiff AUTO-CLEAR
   ✗ Movements persist indefinitely if auto-clear fails
   ✓ 1500ms timeout always fires (unless timer not cleared on unmount)
   Impact: Memory leak possible if cleanup skipped

4. normalizeGameState() IS REQUIRED
   ✗ Storing raw state from SignalR causes inconsistencies
   ✓ Always call normalizeGameState(incomingState, previousState)
   Impact: UI sync issues, stale prompts, ghost entities

5. EVENT LOG PARSING MUST BE CAREFUL
   ✗ Assuming all entries are Phase 3 types crashes handler
   ✓ Check entry.type before dispatching Phase 3 logic
   Impact: Phase 3 events disappear or cause exceptions

6. STORE COUPLING IS INTENTIONAL
   ✗ Trying to use tileOverlayStore as source of truth
   ✓ gameStore is source of truth, others are derived
   Impact: Stale state, race conditions, desyncs

7. MOVEMENT DETECTION ONLY CHECKS FIRST NEIGHBOR
   ✗ Trying to detect all attacking neighbors
   ✓ Break on first match (this is intentional)
   Impact: Multiple simultaneous attacks only show first

8. ZUSTAND STATE IMMUTABILITY
   ✗ Modifying state object directly: state.tiles[key] = newTile
   ✓ Spreading: { ...state.tiles, [key]: newTile }
   Impact: React won't detect changes, UI won't update
