================================================================================
                    4. CROSS-SERVICE DEPENDENCIES & CYCLES
================================================================================

DEPENDENCY FLOW (no circular):
  GameService (facade) 
    → RoomService (owns _rooms, no deps)
    → LobbyService (→ RoomService)
    → GameplayService (→ RoomService + GameStateService)
    → HostControlService (→ RoomService + GameStateService)
    → GameStateService (→ RoomService + RoomPersistenceService)

CRITICAL PATTERN: Modify State + Persist
  1. Get room via RoomService.GetRoom()
  2. Acquire lock on room.SyncRoot
  3. Modify room.State (inside lock)
  4. Create snapshot via GameStateService.GetStateSnapshot()
  5. Queue persistence via GameStateService.QueuePersistence()
  6. Release lock
  7. Return snapshot to caller

NO CYCLES: All dependencies flow downward. GameStateService calls
RoomPersistenceService (async, non-blocking).

⚠️ ASYNC/LOCK HAZARD:
  Release lock BEFORE calling async methods.
  Current code pattern is CORRECT: extract data under lock, release, then
  call async RoomPersistenceService.

✓ THREADING:
  RoomPersistenceService uses IServiceScopeFactory to create Scoped DB contexts
  inside async methods. This is safe (scoped contexts isolated per async call).

================================================================================
                          6. DI/LIFETIME PITFALLS
================================================================================

✓ SINGLETON REQUIREMENT (Critical)
  All 6 services must be AddSingleton (they share _rooms dict in memory)
  This is NON-NEGOTIABLE due to in-memory game state architecture.

✗ PITFALL 1: Scoped Services
  If any service is Scoped, multiple instances = multiple _rooms dicts
  = rooms become invisible to other instances
  = BROKEN GAMEPLAY

✗ PITFALL 2: Holding Locks During Async
  lock (room.SyncRoot) { ... await asyncCall(...); }  ← DEADLOCK RISK
  SOLUTION: Release lock before async calls (current code does this ✓)

✗ PITFALL 3: BG Services Creating New GameService
  TroopRegenerationService creates Scoped container → GetRequiredService<GameService>()
  BUT GameService is Singleton, so even from Scoped container, you get same
  Singleton instance. This is SAFE ✓

✗ PITFALL 4: Static Mutable State
  If Colors[], AllianceColors[], CopresencePresets were mutated concurrently
  = RACE CONDITIONS
  SOLUTION: Make them readonly at init (current code does this ✓)

✗ PITFALL 5: Missing Locks on room.State
  If code modifies room.State without lock = RACE CONDITIONS
  SOLUTION: All public methods must acquire room.SyncRoot lock

================================================================================
                              RECOMMENDATIONS
================================================================================

1. Create comprehensive unit tests for each new service BEFORE splitting
   - Test RoomService room tracking
   - Test GameStateService snapshots & persistence
   - Test LobbyService configuration validation
   - Test GameplayService movement & combat
   - Test HostControlService admin ops

2. Keep GameService facade thin
   - No business logic in facade
   - Just route method calls
   - Makes it easy to refactor further later

3. Consider Future Splitting
   - HostControlService could be merged into GameplayService if needed
   - GameStateService helpers could grow into separate StateTransformService
   - Could eventually have PeristenceService and StateService separate

4. Document Lock Requirements
   - Every public method must document: "Requires caller to hold room.SyncRoot"
   - Or alternatively, method acquires lock internally
   - Be consistent for clarity

5. Monitor Performance
   - TroopRegenerationService iterates GetPlayingRoomCodes() every 30s
   - With 100s of concurrent rooms, this could be slow
   - Consider caching active room codes if needed

6. Consider Event/Message Bus
   - Currently uses lock-based communication
   - Could eventually migrate to message-driven architecture
   - But that's a separate, larger refactor
