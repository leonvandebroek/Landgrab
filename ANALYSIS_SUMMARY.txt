Phase 10: Duel, Hostage

Each mode implements location-triggered game mechanics when players share hexes.

PLAYER ROLES (5 TOTAL):
1. Commander: +1 attack in hex
2. Scout: +2 troops on first hex visit
3. Defender: Double regeneration when present
4. Saboteur: Enemy hexes lose -1 troop/update
5. Engineer: Build forts after 10 min, +1 permanent defense

COMBAT BONUSES (8 TYPES):
1. PresenceBonus: +1 attacker (physically present)
2. Terrain: +1 (Building/Hills) or +2 (Steep) defender
3. Rally: +1 defender (2+ allies in hex)
4. FrontLine: +1 attacker per adjacent ally
5. Commander: +1 attacker (role in hex)
6. Fort: +1 defender (permanent Engineer structure)
7. UnderdogPact: +2 attacker (vs dominant alliance >60%)
8. Timed Escalation: +N regen bonus (1 per 30 min)

TERRAIN SYSTEM (9 TYPES):
Water (impassable), Building (+1 defense, +1 regen), Road, Path, Forest, Park,
Hills (+1 defense), Steep (+2 defense), None

WIN CONDITIONS (3 TYPES):
1. TerritoryPercent: Claim X% (1-100%) of claimable hexes
2. Elimination: Last alliance/player with territory
3. TimedGame: Highest territory after N minutes

ACHIEVEMENTS (4 TYPES, computed on game over):
1. Territory Leader: Highest territory count
2. Army Commander: Most total troops on map
3. Conqueror: Most tiles captured
4. First Strike: Earliest capture event

FEATURE FLAGS (10+ TOGGLES):
TerrainEnabled, PlayerRolesEnabled, FogOfWarEnabled, SupplyLinesEnabled,
HQEnabled, TimedEscalationEnabled, UnderdogPactEnabled, NeutralNPCEnabled,
RandomEventsEnabled, MissionSystemEnabled

================================================================================
                        KEY DESIGN PATTERNS
================================================================================

1. SERVICE FACADE PATTERN
   - GameService as main facade coordinating 19 services
   - Cleaner client interface

2. PROVIDER PATTERN
   - IGameRoomProvider for dependency injection
   - RoomService implements provider

3. IMMUTABLE SNAPSHOTS
   - All state changes create snapshots
   - Prevents corruption, enables async persistence

4. EVENT SOURCING
   - Full GameEventLogEntry audit trail
   - Complete game history available

5. LOCK-BASED SYNCHRONIZATION
   - room.SyncRoot protects all mutations
   - No async during game state changes

6. PER-PLAYER STATE FILTERING
   - GetPlayerSnapshot() for Fog of War
   - Server-side filtering prevents cheating

7. BACKGROUND SERVICE PATTERN
   - MissionService, RandomEventService, TroopRegenerationService
   - Async ticks without blocking game state

================================================================================
                             CONCLUSION
================================================================================

LandGrab backend implements a sophisticated location-based territory control
game with extensive customizable mechanics. The codebase is well-organized
with clear separation of concerns across 19 specialized services, 18 location-
triggered copresence modes, and 5 player roles. Game logic is centralized in
GameplayService (1,147 lines) with proper state management, validation, and
persistence.

The architecture supports complex interactions including:
- Simultaneous 8-faction alliances with shared territory
- 15+ mechanics triggering per player location update
- 12+ systems processing per troop regeneration tick
- Deterministic combat with 8 bonus types
- Supply line connectivity requirements
- Terrain-based defense modifiers
- Per-player visibility filtering (Fog of War)
- Full audit trail of all game events

Test coverage exists for core mechanics but could be expanded. All critical
features are implemented except 4 TODO items (Prey Escape, Relay mode, Event
Warning, NPC faction).
